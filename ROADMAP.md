## 1. Что уже сделано и почему (кратко)

Приложение — PWA для пауэрлифтинга: считает рабочие веса от 1ПМ, ведёт программы, журнал, отчёты.

| Блок | Что | Почему так |
|---|---|---|
| **Калькулятор** | Ввод 1ПМ (присед/жим/тяга) → 4 таблицы недель (Н1–Н4) от 1ПМ; суммы; история | Быстрый экран «сколько вешать». 105% (проходка) убрана — это соревнования, не тренировка |
| **Программы** | Мастер **уровень → дисциплина → упор** → готовая программа от 1ПМ; конструктор по неделям/дням (пилюли, 3 дня, «движение на день») + подсобки к движениям | Пользователь хотел простой выбор и редактирование; дни — чтобы не было «простыни» |
| **Журнал** | ПЛАН + ФАКТ + история по неделям/дням; автопрогрессия (какая тренировка сейчас); вес тела/тайминг/таймер подходов — в тренировке; история редактируется | «Дневник» выполнения; пользователь хотел видеть, где он в цикле, и мерить отдых |
| **Отчёты** | Кол-во тренировок, длительность, частота, объём, качество, отдых/нагрузка, тоннаж | Видеть динамику и нагрузку |
| **Подсобки** | Каталог + «проблема → подсобки» | Подбор вспомогательных под типовые ошибки |
| **Профиль** | ФИО, дата рождения, аватар, вес тела (из последней тренировки) | Упрощён по просьбе: роль/тайминг убраны (тайминг переехал в тренировку) |
| **Админ** | Правка контента: тексты ошибок, связи «проблема→подсобки», каталог подсобок + **базовый шаблон** (матрица недель, базовые подсобки) | Пользователь = админ: контент должен править он, а не код |

**Ключевая архитектура:** контент = **редактируемые данные**, отделён от логики. Дефолты в `config.js`/`content.js`/`accessorydata.js`/`templates.js`; правки админа лежат в `localStorage` поверх дефолтов через `contentstore.js` (`getErrors/getCatalog/getProblemFix/getWeeks/getAccDefaults` + `save*`/`reset*`). Хранилище Фазы 1 — только `localStorage`.

Подробный список решений — в §4. Историю по фазам см. в `TZ.md` §9–11 и в `git log`.

## 2. Что дальше — планируемые фичи (и зачем)

### Приоритет 1 — Фаза 2: тренер ↔ атлет (бэкенд)
- **Что:** авторизация; роли тренер/атлет; тренер видит своих атлетов, их программы/журналы/отчёты; назначение программ атлету; синхронизация между устройствами.
- **Зачем:** сейчас всё в `localStorage` — данные живут на одном устройстве, тренер не видит подопечных. Для реальной работы зала нужен общий доступ и синхронизация.
- **Технология:** Supabase (auth + БД + realtime). Токены/ключи даёт пользователь.
- ⚠️ **Перед стартом ответить на вопросы из §3** (пользователь сам просил их задать).

### Приоритет 2 — улучшения «по факту использования»
- **Отчёты-графики:** прогресс 1ПМ во времени, объём по неделям, динамика веса тела. *Зачем:* видеть тренд, а не только средние.
- **Точечные схемы в админке:** редактировать программу отдельно под каждый «упор × уровень» (сейчас упор влияет на подсобки, уровень — на число подходов). *Зачем:* реальная разница программ, а не косметическая.
- **Экспорт/импорт данных (JSON-бэкап):** кнопка «сохранить всё / загрузить». *Зачем:* не потерять данные при смене устройства или очистке браузера — актуально до Фазы 2.

### Бэклог / идеи (не приоритет)
- Звук/вибро по таймеру отдыха.
- Шаринг программы или отчёта ссылкой.
- Печать/экспорт программы в PDF.
- Тёмная/светлая тема, размер шрифта.

## 3. Открытые вопросы — ОТВЕТИТЬ ПЕРЕД Фазой 2

Пользователь просил задать «ключевые глобальные» вопросы по модели тренер↔атлет. Спросить **до** написания кода:

1. **Привязка атлет↔тренер:** как? (тренер даёт код-приглашение / добавляет атлета по email / атлет выбирает тренера из списка?)
2. **Владение программой:** тренер назначает программу атлету, или атлет берёт сам, а тренер только смотрит? Может ли атлет править назначенную программу?
3. **Видимость:** тренер видит весь журнал/отчёты атлета? Атлеты тренера видят друг друга (скорее нет)?
4. **Двойная роль:** один человек — и тренер, и сам тренируется? (вероятно да)
5. **Вход:** email+пароль / магик-ссылка / соцсети? Нужен ли гостевой режим без регистрации?
6. **Что синхронизируется** (1ПМ, программы, журнал, профиль, вес) и что приватно?
7. **Оффлайн (PWA) остаётся** — как мержить при рассинхроне (последняя запись побеждает?)
8. **Контент админки** (шаблоны/ошибки/подсобки) — общий для всех (один админ — ты) или у каждого тренера свой набор?
9. **Supabase:** чей проект/ключи, какой план (бесплатный tier ок?), регион.

## 4. Принятые решения (чтобы не переспрашивать)

- **PWA на чистом HTML/CSS/JS (ES-модули), без фреймворков/сборки.** Почему: весит килобайты, раздаётся ссылкой/QR, работает на всех платформах, ставится на экран, офлайн, бесплатный хостинг. Unity сознательно отвергнут.
- **Деплой = `git push` в `main`** (GitHub Pages: `main`/root). `CACHE` в `sw.js` поднимать при любой правке файлов — сигнал авто-обновления.
- **Хранилище Фазы 1/1.5 — только `localStorage`.** Бэкенд — Фаза 2 (Supabase).
- **105% не используется в программах** — проходка = соревнования. Максимум 100%. 105% остаётся числовой плиткой «цель цикла» в калькуляторе.
- **Округление до 0,1 кг** (точное, направление выбирает атлет) + подсказка «на снарядах» кратно 2,5.
- **Программы:** уровень→дисциплина→упор → генерация от 1ПМ (`templates.buildTemplate`), редактируемая копия. Модель: `weeks → days (3, движение на день) → exercises {основное: %/повт; подсобки: подходы×повт}`. Старые программы мигрируются (`migrateProgram`).
- **Журнал:** автопрогрессия по числу завершённых тренировок; вес тела, тайминг и таймер подходов — внутри тренировки; пустые вес/повт при завершении = план; история редактируется; сессия хранит самодостаточный `items`-снимок (+ `workSec/restSec/setsTimed`).
- **Контент = данные, отделённые от логики**, правятся в «Админ» через `contentstore` (оверрайды в `localStorage`, сброс к дефолту).
- **Профиль упрощён:** ФИО, дата рождения (возраст считается), аватар, вес тела. Роль и тайминг убраны (роль вернётся в Фазе 2).
- **Контент — черновик:** тексты ошибок/подсобок и схемы заполнены «по логике», пользователь вычитывает/правит через админку.
